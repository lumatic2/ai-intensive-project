# PRD — 생활 맥락 로봇청소기 시뮬레이터

> 본 문서는 PLANNING.md의 시나리오 4종(§4)과 데모 UI(§5)를 user story·acceptance criteria로 변환한다.
> "무엇을 만드는가"에 답한다. "어떻게"는 [TRD.md](./TRD.md).

## 1. 제품 정의

**한 줄.** 사용자가 시나리오를 선택하면, 로봇청소기가 어느 공간을 어떤 순서·모드로 청소할지 결정하고 그 이유를 자연어로 설명하는 웹 시뮬레이터.

**비-목표.**
- 실제 로봇청소기 제어
- SLAM·경로 최적화
- 멀티 디바이스(가전 연동)
- 사용자 계정·로그인·DB

## 2. 핵심 사용자 여정

```
[랜딩] → [집 구조 2D 맵 + 시나리오 4개 버튼 노출]
   ↓ 사용자가 시나리오 클릭 (예: "비 오는 날 귀가")
[백엔드: scoring → LLM 설명 생성] (~3-5초)
   ↓
[맵 heatmap 갱신] + [우선순위 리스트] + [AI 설명 카드]
   ↓ 사용자가 "왜?" 토글 클릭
[공간별 점수 breakdown 표시]
```

## 3. 기능 요구사항

### F1. 집 구조 2D 맵

| 요건 | 상세 |
|---|---|
| F1.1 | 5개 공간(현관·거실·주방·침실·욕실)을 직사각형으로 배치 |
| F1.2 | 각 공간에 현재 priority score 숫자 표시 |
| F1.3 | score 크기에 비례한 heatmap 색상 (낮음=회색, 중간=노랑, 높음=빨강) |
| F1.4 | 제외된 공간은 hatched/다른 패턴으로 시각 구분 |
| F1.5 | 사용자 현재 위치를 아이콘(●)으로 표시 |

**Acceptance.** 시나리오 미선택 시 모든 공간이 base score(중립색). 시나리오 선택 후 색·숫자가 1초 내 갱신.

### F2. 시나리오 선택 패널

| 요건 | 상세 |
|---|---|
| F2.1 | PLANNING.md §4의 4개 시나리오 버튼: "비 오는 날 귀가", "요리 직후", "취침 직전", "손님 방문 예정" |
| F2.2 | 선택 시 백엔드 `/api/simulate` 호출 |
| F2.3 | 로딩 상태 표시 (스피너 또는 skeleton) |
| F2.4 | 에러 시 재시도 버튼 |

**Acceptance.** 동일 시나리오 재클릭 시 캐시 hit으로 즉시 응답.

**확장(P2).** "직접 입력" 모드 — 시간/날씨/이벤트를 폼으로 받아 임의 시나리오 생성.

### F3. 청소 우선순위 리스트

| 요건 | 상세 |
|---|---|
| F3.1 | 점수 내림차순 정렬 |
| F3.2 | 각 항목: 공간명·최종점수·청소모드 뱃지(일반/저소음/지연/제외) |
| F3.3 | 제외된 공간은 회색 처리 + 제외 사유 한 줄 |

**Acceptance.** 시나리오 1(비 오는 날 귀가) 클릭 시 순서가 [현관, 거실, 주방, 침실(제외)] 로 표시.

### F4. AI 설명 카드

| 요건 | 상세 |
|---|---|
| F4.1 | LLM이 생성한 자연어 설명 본문 |
| F4.2 | "왜 이 공간을 먼저?" 토글 — 클릭 시 점수 breakdown(기본/이벤트/페널티) 표시 |
| F4.3 | "왜 이 공간을 제외?" 토글 — 제외 사유 |
| F4.4 | LLM 호출 실패 시 fallback 메시지: "AI 설명을 가져오지 못했습니다. 점수 근거는 아래 표를 참고하세요." |

**Acceptance.** 출력에 점수 근거(숫자)가 95% 이상 포함 (PLANNING.md §11 KPI).

## 4. 비기능 요구사항

| 항목 | 목표 |
|---|---|
| 응답 속도 | 시나리오 클릭 → 화면 완전 갱신 5초 이내 |
| 일관성 | 동일 입력 반복 시 우선순위 100% 동일 (Rule-based) |
| 데모 안정성 | 발표 중 시연 성공률 100% — 모든 시나리오 응답 사전 캐싱 |
| 모바일 대응 | 발표는 노트북·프로젝터 기준. 모바일 미지원 |
| 다국어 | 한국어만 |

## 5. 우선순위

| Priority | 범위 |
|---|---|
| **P0 (1주차 끝)** | F1.1~F1.3, F2.1~F2.3, F3.1~F3.3, F4.1, 시나리오 1~2 작동 |
| **P1 (2주차 끝)** | F1.4~F1.5, F4.2~F4.4, 시나리오 3~4, LLM 캐싱 |
| **P2 (여유 시)** | F2 직접 입력 모드, ML 이벤트 분류기 화면 노출, 누적 패턴 요약 |

## 6. 성공 기준 (PLANNING.md §11에서 발췌·매핑)

| KPI | 측정 방법 |
|---|---|
| 작동 시나리오 수 ≥ 4 | 시나리오 4개 모두 정상 응답 |
| 응답 속도 < 5초 | 브라우저 DevTools Network 탭 |
| 일관성 100% | 동일 시나리오 10회 호출 → 우선순위 동일 |
| 설명 품질 (점수 근거 포함률) ≥ 95% | 시나리오 4개 × 5회 = 20회 출력 수동 채점 |

## 7. 미해결·후속 결정

- 시나리오별 사전 캐싱 응답을 git에 commit할지 (저장소 크기 vs 발표 안정성)
- ML 이벤트 분류기 결과를 화면에 노출할지 (P2 — 발표 슬라이드 6번 자료로만 활용 vs UI 통합)
- 욕실을 5번째 공간으로 둘지(추가) — PLANNING.md 표는 4개 공간만 다룸. 결정 필요
